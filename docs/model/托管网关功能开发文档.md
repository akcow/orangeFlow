# 托管网关（Hosted Gateway）功能开发文档（MVP 优先）

## 1. 背景与问题

当前 LangFlow 的组件密钥配置页可以统一管理各厂商的 API Key，但用户仍然需要：

1) 自己去每家厂商注册/申请 Key  
2) 在 Settings 里逐个填写  
3) 处理不同厂商的模型命名、计费、限流与错误差异

这对新用户上手成本很高，也不利于我们后续做统一的稳定性、审计、用量统计与商业化。

## 2. 目标（用户视角）

**核心目标：用户只需要一个“我们发的 Key”，就能使用现有所有厂商的所有模型（包含图片/视频）。**

用户体验要求：
- Settings 里只配置 1 个 Key（或登录后自动有）
- 在选择模型时只关注“模型名/能力/价格/延迟”等，不再需要理解厂商 Key
- 错误提示一致、可定位；流式体验稳定

## 3. 非目标（第一阶段不做）

为保证快速迭代，以下能力在 MVP 暂不强制：
- 完整覆盖所有 OpenAI API 变体（Responses API、Realtime、Audio 等）
- Agent / tool_call（当前自定义组件未依赖，可后置）
- 复杂的策略路由（AB test、按地域多活、智能降级）
- 精确到毫分的计费（先用近似 token 计费即可）
- 企业级合规全套（先保证默认不落 prompt 内容 + 可扩展审计）

## 4. 总体方案（一句话）

在现有 LangFlow 后端中新增一个 **Hosted Gateway**：主要供 LangFlow 自己调用；对外暴露统一接口（优先 OpenAI-compatible），对内按 `model` 路由到不同厂商适配器；厂商 Key 由我们托管；对用户只暴露“我们的 Key”。

本项目的特殊约束（已确认）：
- 需要覆盖“现有所有厂商的所有模型”，并包含图片/视频生成能力
- Gemini 与 Sora 系列共用一套国内代理（见“国内代理支持”）
- Hosted Gateway Key 按 user 维度发放
- MVP 允许单实例（限流/配额可先做进程内）

> 迭代策略：先集成在现有后端（快），功能稳定后可按需拆成独立服务（隔离与扩缩容更好）。

## 5. MVP 范围（必须可见效果）

### 5.1 网关 API（MVP）

优先以 OpenAI-compatible 作为“统一入口”，但会补齐图片/视频等现有能力所需的端点：

- `GET /v1/models`
  - 返回可用模型列表（我们定义的 `provider/model` 命名空间 + 能力元数据）
- `POST /v1/chat/completions`
  - 文本/多模态（含图片输入）/结构化输出（JSON）
  - 支持非流式与流式（SSE）
- `POST /v1/images/generations`
  - 图片生成（覆盖现有图片模型）
- `POST /v1/videos`
  - 视频生成（长任务创建，覆盖 Sora、Doubao Video、Wan、Veo 等）
- `GET /v1/videos/{id}`
  - 视频任务状态查询（轮询）
- `POST /v1/audio/speech`
  - 语音合成（TTS，覆盖 Qwen TTS 等）

> 覆盖“现有所有模型”意味着 MVP 必须以“能力分层 + 白名单开关”的方式交付：网关骨架先就位，模型清单以配置/注册表为准；逐批放量，失败模型在 `/v1/models` 中不可见或标记为不可用。

### 5.2 模型命名与路由（MVP）

统一用 `provider/model` 命名（示例）：

**文本模型 (Text):**
- `deepseek-chat`
- `deepseek-reasoner`
- `gemini-3-pro-preview`
- `gemini-3-flash-preview`

**图片模型 (Image):**
- `doubao-seedream-4-5-251128` (Seedream 4.5)
- `doubao-seedream-4-0-250828` (Seedream 4.0)
- `gemini-2.5-flash-image` (Nano Banana)
- `gemini-3-pro-image-preview` (Nano Banana Pro)
- `wan2.6-t2i` / `wan2.6-image`
- `wan2.5-t2i-preview` / `wan2.5-i2i-preview`

**视频模型 (Video):**
- `doubao-seedance-1-5-pro-251215`
- `doubao-seedance-1-0-pro-250528`
- `doubao-seedance-1-0-pro-fast-251015`
- `wan2.6`
- `wan2.5`
- `veo-3.1-generate-preview`
- `veo-3.1-fast-generate-preview`
- `sora-2`
- `sora-2-pro`

**音频模型 (Audio):**
- `qwen3-tts-flash-2025-11-27`

路由规则（MVP）：
- 解析 `model` 的 `provider` 前缀，选择对应适配器
- 适配器负责把通用字段映射成厂商字段，并把响应映射回统一结构
- 对每个 endpoint 做能力校验：例如视频模型不能走 `chat/completions`，应返回稳定错误码（见“错误规范”）

### 5.3 兼容边界（MVP，避免“看似兼容但不可用”）

由于“全厂商 + 全模型”能力差异极大，MVP 明确约束如下：

- `POST /v1/chat/completions`
  - 支持 `messages`：文本与图片输入（`image_url`/base64）
  - 支持 `response_format` 的 JSON 约束：
    - 若上游原生支持则透传
    - 若上游不支持则降级为提示词约束，并在响应元数据中标记 `json_mode=degraded`
  - 支持 `stream=true`（SSE）；客户端断连时可取消上游请求
  - 不承诺 `tool_calls/function_calling`（后置）

- `POST /v1/images/generations`
  - 只承诺“生成结果可用”（URL 或 base64）
  - 不强制 OpenAI 图片响应的全字段对齐

- `POST /v1/videos` + `GET /v1/videos/{id}`
  - 采用“创建 + 轮询”模式，便于覆盖长任务
  - 优先对齐现有 Sora 国内代理接口形态（见“国内代理支持”）

### 5.4 国内代理支持（Gemini / Sora）

现有代码中已存在国内代理调用路径（供网关适配器复用/对齐）：

- Gemini（`v1beta generateContent`）
  - 代理基址存在两套默认值：`https://cdn.12ai.org/v1beta`、`https://new.12ai.org/v1beta`
  - 建议网关统一为可配置项（例如按环境变量/配置文件注入），避免组件各自硬编码

- Sora（视频）
  - 默认基址：`https://cdn.12ai.org`
  - 通过 `POST /v1/videos` 创建任务、`GET /v1/videos/{id}` 轮询
  - 支持可选查询参数：`group`、`distributor`

### 5.5 鉴权（MVP）

对外/对 LangFlow 调用：
- Header：`Authorization: Bearer <HOSTED_GATEWAY_KEY>`
- 失败：401 + 统一错误体（见“错误规范”）

对内：
- 使用我们托管的厂商 Key（从 secrets/环境变量/密钥库读取）

## 6. 组件与代码结构建议（防止代码膨胀）

建议按“核心网关 + 适配器插件”拆分，新增厂商只新增一个模块：

- `gateway/router.py`
  - 解析 `model`，选择适配器
- `gateway/schemas.py`
  - 统一请求/响应的数据结构（内部标准，按 chat/image/video 分层）
- `gateway/providers/base.py`
  - `ProviderAdapter` 抽象接口
- `gateway/providers/openai.py`, `anthropic.py`, `gemini.py`, `deepseek.py`, `sora.py`, ...
  - 仅实现字段映射与该厂商特殊逻辑（含国内代理基址、轮询协议等）
- `gateway/http.py`
  - 统一的超时、重试、连接池、代理、TLS、观测埋点
- `gateway/auth.py`
  - Hosted Key 验证（与 LangFlow 现有 auth 复用/衔接）
- `gateway/errors.py`
  - 统一错误码与映射

> 原则：核心层绝不出现“按厂商写一堆 if/else”；所有差异都收敛在 providers 目录里。

## 7. 数据与密钥设计（从简到全）

### 7.1 我们对用户发放的 Hosted Gateway Key

MVP 方案（快速上线）：
- 每个用户生成一个 Hosted Key
- Hosted Key 只用于调用网关，不具备 LangFlow 管理权限（与 LangFlow API key 区分）

推荐字段（数据库或配置存储）：
- `key_id`（可公开）+ `key_hash`（只存 hash，不存明文）
- `owner_type`/`owner_id`（user）
- `status`（active/revoked）
- `created_at`/`revoked_at`
- `rate_limit`（QPS/并发，后续启用）
- `monthly_quota`（后续启用）

### 7.2 厂商 Key 托管

MVP 方案：
- 厂商 key 不进数据库，先以部署环境变量或 secrets 文件方式托管
- 默认不记录 prompt/response 内容，仅记录用量与元数据（见“观测与日志”）

（按现有组件/配置习惯）常见 key 例子：
- Gemini：`GEMINI_API_KEY` / `GOOGLE_API_KEY`
- Sora：`OPENAI_API_KEY`

后续增强：
- 接 KMS/Vault/云密钥管理
- 轮换与多 key 池（提升限流与稳定性）

## 8. 错误规范（统一对外体验）

对外统一错误体（示例）：

```json
{
  "error": {
    "code": "PROVIDER_RATE_LIMITED",
    "message": "Upstream provider rate limited. Please retry later.",
    "provider": "anthropic",
    "request_id": "req_..."
  }
}
```

约定：
- `code`：我们自己的稳定错误码（不要直接透传上游错误码）
- `provider`：便于排查（可选）
- `request_id`：贯穿全链路（必须）

建议新增（MVP 常用）：
- `MODEL_NOT_FOUND`
- `MODEL_NOT_SUPPORTED_FOR_ENDPOINT`
- `PROVIDER_AUTH_FAILED`
- `UPSTREAM_TIMEOUT`

## 9. 观测与风控（MVP 也必须有的最小集合）

### 9.1 结构化日志（默认不落内容）

每次请求至少记录：
- `request_id`
- `owner_id`（或 key_id）
- `endpoint`（chat/images/videos）
- `model`、`provider`
- `status_code`、`error_code`（若失败）
- `latency_ms`
- `prompt_tokens`、`completion_tokens`、`total_tokens`（能拿到就记）

视频等长任务额外记录（如有）：
- `task_id`
- `poll_count`/`generation_time_s`

默认不记录：
- prompt/messages 原文
- tool 参数与输出原文

### 9.2 限流与保护（迭代启用）

先做简单保护即可，快速提升用户体验稳定性：
- 按 Hosted Key 的并发限制（避免单 key 打挂系统）
- 统一上游超时（防止请求堆积）
- 仅对“明确可重试”的错误做重试（避免雪崩）

## 11. 与 LangFlow 集成方式（建议）

MVP 集成建议：
- 网关 API 与现有 FastAPI 同进程部署（复用配置、日志、认证基础设施）
- 通过环境变量开启/关闭 Hosted Gateway（便于灰度）

拆分时机（建议）：
- 当开始做：商业计费、多租户隔离、独立扩缩容、对外 SLA 时
- 将 gateway 抽成独立服务，LangFlow 后端作为控制面（发 key、配额、路由规则），gateway 作为数据面（高 QPS 转发）

## 12. 安全与合规检查清单（上线前）

- [ ] 默认不记录 prompt 内容（可配置的审计模式需明确提示）
- [ ] secrets 不落库/不落日志；日志中对 token/key 做脱敏
- [ ] request_id 全链路贯通
- [ ] 明确的超时、重试与最大响应大小限制（避免 DoS）
- [ ] Hosted Key 可吊销、可轮换

## 13. 附：调用示例（MVP）

### 13.1 非流式

```bash

  -H "Authorization: Bearer $HOSTED_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deepseek-chat",
    "messages": [{"role": "user", "content": "你好，介绍一下你自己"}]
```

```bash
curl https://$HOST/v1/chat/completions \
```

## 14. 测试与发布（快速但可控）

### 14.1 测试清单（按优先级）

P0（必须过）：
- [ ] Hosted Key 鉴权：缺失/错误 key 必须 401，且不泄露细节
- [ ] 模型覆盖：`/v1/models` 中所有标记可用的模型都可调用成功（按其 endpoint）
- [ ] 流式成功：SSE 不断流；中途断开能被正确取消/释放资源
- [ ] 统一错误：上游 429/5xx/超时 都映射到稳定错误码
- [ ] 日志脱敏：日志中不出现任何 key/token
- [ ] 多模态：图片输入可用（若模型不支持，返回稳定错误码）
- [ ] 结构化输出：JSON 约束生效（或明确降级策略）
- [ ] 图片生成：至少 1 个图片模型可用
- [ ] 视频生成：创建 + 轮询可用（至少 1 个视频模型可用）

P1（建议过）：
- [ ] 超时与重试：不会导致重复扣费/重复输出（流式尤其注意）
- [ ] 并发压测（轻量）：同一 key 并发 N（例如 10）不崩溃
- [ ] 模型列表：`/v1/models` 输出与实际路由一致

### 14.2 灰度与回滚

灰度方式（建议）：
- 用环境变量开关 Hosted Gateway 路由（例如仅对部分内部 key 开启）
- 先只开放少量模型与小配额

回滚策略：
- 一键关闭网关入口（返回 404/503），不影响 LangFlow 其他功能
- 保留旧的“用户自带 key”路径作为兜底（可选）
